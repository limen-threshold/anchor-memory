# v1.13 — Stepped History Trimming (cache-friendly window bounding)

One lesson from production, generalized: **a per-turn sliding window silently defeats provider prompt caching.** "Keep the newest N pairs, every turn" drops one pair off the *front* each turn — the prompt prefix changes at byte 0, and providers with prefix caching (Anthropic explicit `cache_control`; OpenAI / DeepSeek implicit) re-bill the whole history at full price every single turn. Measured in the production system this mirrors: a ~5× cost difference on a large window.

## New: `--history-pairs N --history-pairs-max M` on anchor_proxy

Hysteresis instead of sliding: the window grows **append-only** from N to M pairs (stable prefix → cache hits every turn), then cuts back to N in one stroke. You pay one full cache rewrite per (M−N) turns instead of every turn.

Tuning rules (docs/cross-window.md § design notes):
- Keep M's token size **below your model's long-context degradation knee** — nominal window ≠ usable window (frontier models show measurable multi-fact-reasoning degradation well before their advertised limit).
- Pick N so the *average* window (≈ (N+M)/2) is the size you actually want.
- Counterintuitive corollary: a bigger window that caches is much cheaper than a smaller one that doesn't.

Default remains 0/0 (no trimming — your client controls the window). `M ≤ N` behaves like the classic sliding window.
